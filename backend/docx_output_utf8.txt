Shop Radar

Social Media Module

Technical Specification & Implementation Guide

Production-Grade | Clean Architecture | Scalable

1. Project Overview

This document describes the complete technical design and implementation plan for integrating a production-grade social media module into the existing Shop Radar Flutter application. The system is modeled after Instagram and includes Posts, Stories, Reels, a Follow graph, real-time Messaging, and push Notifications.

1.1 Core Objectives

Allow all users to create and consume content: posts, reels, and stories

Support a bidirectional social graph (follow / unfollow)

Enable real-time one-to-one messaging via Socket.io

Deliver push notifications exclusively through Firebase Cloud Messaging (FCM)

Treat shops as a special user type (accountType: shop) without breaking existing data

Maintain clean, modular, and scalable code across frontend and backend

2. System Architecture

2.1 Backend Layer Pattern

Every backend feature follows the Controller ΓåÆ Service ΓåÆ Repository (CSR) pattern:

Router receives the HTTP request and delegates to the Controller

Controller validates input (using Joi or class-validator), calls the Service, and formats the HTTP response

Service contains all business logic (auth checks, notification triggers, count updates)

Repository is the only layer that touches MongoDB. It returns plain objects to the Service

3.3 JWT Implementation Notes

Access token TTL: 15 minutes

Refresh token TTL: 30 days ΓÇö stored in DB or Redis

All protected routes require Authorization: Bearer <accessToken> header

On 401, Flutter client silently refreshes and retries the original request

FCM token must be updated on every successful login

3.4 Password & Security

Input validation: username (alphanumeric + underscore, 3-30 chars), password (min 8 chars)

Rate limiting on /auth/* endpoints (express-rate-limit: 10 req / 15 min per IP)

4.3 Atomic Count Updates

When a follow is created or deleted, use MongoDB $inc to update both counters in a single operation. Never derive counts by counting documents at query time.

// On follow

await User.updateOne({ _id: followingId }, { $inc: { followersCount: 1 } });

await User.updateOne({ _id: followerId },  { $inc: { followingCount: 1 } });

// On unfollow ΓÇö decrement, floor at 0

await User.updateOne({ _id: followingId }, { $inc: { followersCount: -1 } });

await User.updateOne({ _id: followerId },  { $inc: { followingCount: -1 } });

5.5 Feed Query Logic (Aggregation Pipeline)

The feed uses a MongoDB aggregation pipeline to avoid N+1 lookups:

Match posts where userId is in the array of followingIds

Sort by createdAt descending

Apply cursor-based pagination (filter by _id < lastSeenId)

$lookup to join user info (username, profilePic, accountType)

$lookup to join the current user's like status on each post

Project only the fields needed by the frontend

5.6 Media Upload Flow

Flutter compresses image/video locally (flutter_image_compress / ffmpeg_kit_flutter)

Flutter POSTs the file as multipart/form-data to /posts/create

Backend receives file via multer (memory storage), streams to Cloudinary

Cloudinary returns a secure URL; backend stores only this URL in MongoDB

Backend responds with the created post object

6.3 Feed Grouping Logic

The /stories/feed endpoint returns stories grouped by userId so the frontend can render one avatar bubble per user. Use MongoDB aggregation:

Match userId IN followingIds AND expiresAt > now

Sort by createdAt ascending

$group by userId, pushing all story documents into an array

$lookup to join user profile data (username, profilePic)

Sort groups: users with unseen stories appear first

8.5 Socket Authentication

The Socket.io server authenticates the connection using a middleware that reads the JWT from handshake.auth.token and attaches the decoded user to socket.data.user. Unauthenticated sockets are disconnected immediately.

8.6 Flutter Chat Implementation

ChatListScreen: StreamBuilder / StateNotifier fed by receive_message events

Optimistic UI: append message to local list immediately, confirm on receive_message echo

Seen receipts: mark_seen emitted when ChatScreen is foregrounded with unread messages

9.4 FCM Dispatch Flow

Triggering action occurs (e.g., user A likes user B's post)

Service layer calls notificationService.send(receiverId, payload)

Notification service fetches receiver's fcmToken from the Users collection

If token exists, calls Firebase Admin SDK messaging().send(message)

Handle TokenNotRegistered error by clearing stale token from DB

9.5 Flutter FCM Setup

Initialize firebase_messaging in main.dart before runApp()

Request notification permissions on iOS (no-op on Android)

FirebaseMessaging.onMessage: show in-app banner (flutter_local_notifications)

FirebaseMessaging.onMessageOpenedApp: navigate to referenceId screen

FirebaseMessaging.onBackgroundMessage: top-level static handler function

Send updated token to PUT /users/update-fcm on every app launch

11. Feed System ΓÇö Detailed Design

11.1 Pagination Strategy

Use cursor-based pagination (keyset pagination) rather than page/offset. This prevents duplicate or missing items when new posts are inserted between page loads.

Request: GET /posts/feed?limit=10&cursor=<lastPostId>

Query: find posts where _id < cursor AND userId IN followingIds

Response includes nextCursor (the _id of the last returned post)

Flutter FeedScreen uses a ScrollController to detect end-of-list and fetch next page

11.2 Following-Feed Aggregation

Auth middleware provides req.user._id

Follow repository returns array of followingIds for req.user._id

Post aggregation: match userId $in followingIds, sort _id desc, apply cursor limit

$lookup users collection ΓåÆ join author info

$lookup likes collection ΓåÆ join isLiked boolean for current user

$project to select only needed fields (avoid sending full user documents)

11.3 Empty State & First-Run

If followingIds is empty, return a discovery feed: recent public posts sorted by likesCount

Flutter renders an EmptyFeedWidget with a "Find people to follow" CTA

Pull-to-refresh resets the cursor and re-fetches from the top

12.2 N+1 Query Prevention

Never fetch posts then loop to fetch each author ΓÇö always use $lookup in the pipeline

Batch follow-status checks in a single $lookup, not per-post

Use .lean() in Mongoose to return plain JS objects (faster than Mongoose documents)

Denormalize lastMessage into the Conversation document to avoid an extra query for chat list

12.3 Future Scalability Options

Redis cache: Cache followingIds list per user (TTL 60s) ΓÇö reduces DB reads on every feed request

Bull / BullMQ: Move notification dispatch to a background job queue

MongoDB Atlas Search: Full-text user and post search via Lucene indexes

CDN: Cloudinary already serves assets from a global CDN ΓÇö no additional setup needed

14. Backward Compatibility & Migration

14.1 Existing Shop Data

The existing Shop Radar database has shop documents with a shopId field. The migration maps these to the new user schema without data loss.

14.2 Migration Steps

Run a one-time migration script: for each Shop document, create a corresponding User document with accountType: "shop" and userId = shopId

Migrate existing shop posts: set userId = shopId (rename field, keep _id)

Add a shopRef field on User if the original Shop document needs to remain for shop-specific data (hours, address, categories)

After migration, update all backend queries that reference shopId to use userId

Deploy backend changes before Flutter changes to maintain backward compatibility during rollout

14.3 Dual-Type UI

Profile screens for accountType: "shop" display shop-specific info (address, hours) below the standard social profile header

Feed and reels treat shop posts identically to user posts

Shop users cannot send direct messages unless enabled (feature flag in User schema)
